Component reference

<adressevaelger-search> is a Lit 3 element. Import adressevaelger-enhanced (or adressevaelger-enhanced/component) to register it.

Attributes & properties

Attribute Property Type Default Description
token token string — API access token (required to search).
base-url baseUrl string https://adressevaelger.dk Override the API base URL.
include-preliminary includePreliminary boolean false Include preliminary (not-yet-finalized) addresses.
placeholder placeholder string Søg efter adresse… Input placeholder text.
max-results maxResults number 10 Maximum suggestions shown.
value value string '' Current input text (read/write).

Events

Event Detail When
adresse-selected { suggestion, address: FullAddress | null, houseNumber: HouseNumber | null } A suggestion is chosen. The picked item is resolved by id (cached); adresse suggestions fill address, husnummer suggestions fill houseNumber.
adresse-input { value: string } On every debounced (250 ms) query change.

Accessibility

  • Implements the WAI-ARIA 1.2 combobox pattern: the input is role="combobox" with aria-autocomplete="list", aria-expanded and aria-controls pointing at the popup role="listbox".
  • Keyboard: ↓/↑ move (and wrap) the active option via aria-activedescendant, Enter selects, Esc closes.
  • The matched portion of each suggestion is wrapped in <mark>; a visually-hidden aria-live region announces result counts.

Headless API reference

createAddressSearch(options) returns a client with its own request caches. It has no DOM dependency — use it in any framework or on the server. 

import { createAddressSearch } from 'adressevaelger-enhanced/lib'

const search = createAddressSearch({
  token: 'your-token',        // required (>=10 chars today; see terms)
  baseUrl: 'https://adressevaelger.dk', // optional
  includePreliminary: false,  // optional
  maxResults: 10,             // optional
})

const controller = new AbortController()
const results = await search.search(controller.signal, 'askevænget 9 2 tv')
// -> AutocompletedAddress[] ranked by relevance, at most maxResults

const full = await search.getAddressById(results[0].id)   // FullAddress | null
const hn = await search.getHouseNumberById('...')          // HouseNumber | null
search.clearCache()

Ranking utilities are exported too: rankAddressResults, scoreAddress, tokenize, tokenMatchScore, hasUnmatchedQueryTokens.

React

import { useMemo, useState } from 'react'
import { createAddressSearch } from 'adressevaelger-enhanced/lib'

function useAddressSearch(token) {
  const client = useMemo(() => createAddressSearch({ token }), [token])
  const [results, setResults] = useState([])
  async function onChange(term) {
    const controller = new AbortController()
    setResults(await client.search(controller.signal, term))
  }
  return { results, onChange }
}

Vue

import { ref } from 'vue'
import { createAddressSearch } from 'adressevaelger-enhanced/lib'

const client = createAddressSearch({ token: 'your-token' })
const results = ref([])
async function onInput(term) {
  const controller = new AbortController()
  results.value = await client.search(controller.signal, term)
}

Or drop the web component straight into a Vue template — custom elements need compilerOptions.isCustomElement configured.

How the ranking works

The search API only filters on street / house number / postal code and orders results by postal code, not relevance — and it ignores floor, door and city in the query. The library re-ranks every candidate's display string (titel) against the raw query.

  1. Tokenize both query and titel (lowercase; commas and periods split; "st. tv" → ["st","tv"]).
  2. Token scoring with Danish-letter folding (aa↔å, ae↔æ, oe↔ø): exact = 1.0, prefix = partial credit growing with coverage. A lone digit never prefix-matches a 4-digit postal code (so "2" reads as a floor, not the start of 2830).
  3. Greedy match each query token to its best unused titel token; the score is average query coverage plus a small titel-coverage term (favouring the shorter / more exact titel among equal matches).
  4. DAWA same-street tie-break: equal scores on the same street sort by house number, then floor (kl < st < 1 < …), then door (tv < mf < th < numeric), then postal code — so a street-only query spreads low house numbers across postal areas the way DAWA did.

Expansion pipeline

query "askevænget 9 2 tv"
      │
      ▼
  /adresser/soeg  ──►  husnummer "Askevænget 9, …"   (floor/door unmatched)
      │                        │
      │         hasUnmatchedQueryTokens? ──► search titel ──► adresse units
      │
  street-only? navngivenvejpostnummer ──► search "vejnavn postnr" ──► husnumre
      │                                          │
      │                              (their own unmatched units expand too)
      ▼
  dedupe by id  ──►  rankAddressResults(query)  ──►  slice(maxResults)

A street + city query ("askevænget vejle") that the API can't match triggers a concurrent fallback: every trailing-word truncation is searched in one wave and the longest match kept, then ranked against the original query so the intended city surfaces first. All lookups share an LRU promise cache, so duplicate and in-flight requests are coalesced.

Benchmark methodology

The dashboard is fed by an hourly GitHub Action running scripts/benchmark.mjs. Each run times three scenarios over a fixed ~10-query set, 3 iterations each, sequentially with ~100 ms gaps to stay polite:

  • dawa — one api.dataforsyningen.dk/adresser?autocomplete=true&per_side=8 request.
  • adressevaelgerRaw — one /adresser/soeg request.
  • adressevaelgerFull — the full library pipeline (cache cleared per query), counting the requests it issues.

Each run appends one JSON line to history.jsonl on the benchmark-data branch (the "database" is a git branch). DAWA failures are recorded as nulls + a failure count rather than crashing — DAWA is scheduled to shut down on 2026-08-17, and the history is meant to capture that transition.
